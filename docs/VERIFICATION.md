# What is the verification system for?

The verification system checks rajac against OpenJDK behavior.
Its main job is to catch regressions in emitted class files for valid inputs and regressions in diagnostics for invalid inputs.

The valid-source path compares rajac output with reference `.class` files produced by `javac`.
The invalid-source path compares rajac diagnostics with reference compiler errors captured from `javac`.

# What does verification compare for valid sources?

Valid verification inputs live under `verification/sources`.
For each source file, the verification runner compiles the source with rajac and compares the resulting class files with the OpenJDK reference output in `verification/output/openjdk_21`.

The comparison is done on pretty-printed class file structure rather than raw bytes.
This keeps the comparison readable and makes bytecode mismatches easier to diagnose.

# What does verification compare for invalid sources?

Invalid verification inputs live under `verification/sources_invalid`.
They are grouped by stage such as `lexer`, `parser`, and `typecheck`.

The verification runner compares rajac diagnostics against the reference errors captured in `verification/output/openjdk_21/invalid/errors.txt`.
The primary compatibility requirement is that rajac reports the same failing source line as OpenJDK.
When rajac intentionally provides a better error message, the verification runner can compare against a rajac-specific override instead of the OpenJDK text.

# Where is the verification infrastructure implemented?

The main verification runner is `crates/verification/src/verification_main.rs`.
That file is the source of truth for:

- compiling valid verification sources with rajac
- comparing rajac and OpenJDK class files
- verifying invalid-source diagnostics
- maintaining error-message overrides
- maintaining any explicitly ignored class-file mismatches

The OpenJDK reference outputs are generated by `verification/compile.sh`.
That script recompiles all valid verification sources with `javac` and regenerates the invalid reference error log.

# How is the verification data laid out?

The verification directory is organized by role:

- `verification/sources`: valid Java sources that should compile and match OpenJDK bytecode
- `verification/sources_invalid`: invalid Java sources that should produce diagnostics
- `verification/output/openjdk_21`: checked-in OpenJDK reference output
- `verification/output/rajac`: temporary rajac output generated by the verification runner

Within `verification/sources`, group files by topic such as `arithmetic`, `controlflow`, `equality`, `algorithms`, or `simple`.
Keep packages and folders aligned so the resulting class file paths are stable and predictable.

# How do I run verification?

Run the verification suite with:

```bash
cargo run -p rajac-verification --bin verification
```

This compiles the valid fixtures with rajac, compares them against the OpenJDK references, then verifies invalid fixtures and prints a summary.

When finishing a unit of work, also run:

```bash
./scripts/check-code.sh
```

# How do I regenerate the OpenJDK reference outputs?

When you add or change verification fixtures, regenerate the reference outputs with:

```bash
./verification/compile.sh
```

This script rewrites `verification/output/openjdk_21`.
After regenerating references, run the verification suite again to confirm rajac still matches the updated reference output.

# When should I add verification fixtures?

Add verification fixtures when a change affects:

- emitted bytecode shape
- control flow lowering
- method invocation lowering
- constant handling
- parser acceptance or rejection
- semantic diagnostics

Verification fixtures are especially valuable when the change should match OpenJDK closely enough to compare exact bytecode structure.

# How should I write valid verification fixtures?

Prefer small, focused examples.
Each file should exercise a specific behavior with as little unrelated language surface as possible.

Choose package names and class names that describe the feature under test.
Keep the source readable enough that a class-file mismatch can be traced back to the fixture without extra context.

When possible, write fixtures that are stable across compiler implementations.
If a fixture accidentally depends on a code-shape optimization difference, simplify it before introducing ignored mismatches.

# How should I write invalid verification fixtures?

Each invalid file should usually demonstrate one failure mode.
Place it in the stage-oriented folder that best matches the intended failure.

The goal is not only that rajac fails, but that it fails in a way that is comparable to OpenJDK.
That usually means keeping the file minimal so the expected failing line is obvious.

# How should I handle better rajac diagnostics than OpenJDK?

Use the override table in `get_error_message_overrides()` inside `crates/verification/src/verification_main.rs`.
Add an override when rajac intentionally produces a better message than OpenJDK and you still want verification to enforce compatibility.

Follow these rules:

1. Keep the OpenJDK line number as the compatibility anchor.
2. Compare the message text against rajac's improved diagnostic.
3. Keep the override specific to the fixture file name.

# When is it acceptable to ignore a class-file mismatch?

Only ignore a mismatch when the bytecode difference is known, intentional, and not worth blocking unrelated verification signal.
Ignored mismatches belong in `get_ignored_class_file_mismatches()` in `crates/verification/src/verification_main.rs`.

This should be the exception, not the default.
Prefer fixing rajac or simplifying the fixture before adding an ignore entry.

# What contributor workflow should I follow for verification changes?

For verification-related work, use this sequence:

1. Add or update fixtures under `verification/sources` or `verification/sources_invalid`.
2. Regenerate OpenJDK reference output with `./verification/compile.sh` if fixture expectations changed.
3. Run `cargo run -p rajac-verification --bin verification`.
4. Run `./scripts/check-code.sh`.
5. Commit both the fixture sources and any updated checked-in reference outputs.

Do not duplicate implementation details that are better kept in code comments unless they are needed for contributors to use the system correctly.
